# ADR-004: Clipboard Integration — arboard vs clipboard vs fallback

**Status**: Accepted
**Date**: 2026-05-18
**Decider**: Morgan (Solution Architect) — approved by franci

---

## Context

US-08 (Clipboard URL Copy, slice-05) requires that pressing `c` in the detail panel copies the
repository URL to the system clipboard. The requirement has an explicit fallback:

> "If clipboard is unavailable (SSH/headless): show URL as plain text with 'select manually'
> note — no crash."

franci uses the tool on:
- macOS (primary — Darwin 25.4.0, aarch64-apple-darwin / x86_64-apple-darwin)
- NixOS/devenv (Linux — via devenv.nix)
- Potentially SSH sessions into remote machines

The clipboard interaction is an external dependency on an OS service that can lie:
- macOS pasteboard: generally reliable; can fail if the process has no display context.
- X11 (Linux): requires a running X server; headless CI or SSH sessions typically have none.
- Wayland (Linux): requires `wl-clipboard` or compositor support.
- SSH: no clipboard access by default unless using clipboard forwarding (SSH -Y or tmux).

The design must satisfy:
1. Works correctly on macOS (primary environment).
2. Does not crash on environments where clipboard is unavailable.
3. Provides a useful fallback (show URL as selectable text).
4. Includes a `probe()` implementation per the Earned Trust principle (brief.md).

---

## Decision

**Use the `arboard` crate** (latest stable version at time of slice-05 implementation).

`arboard` provides:
- macOS: native NSPasteboard via Objective-C bindings — the correct macOS clipboard API.
- Linux: X11 (via `x11-clipboard`) and Wayland (via `wl-clipboard-rs`) with automatic selection.
- Windows: Win32 clipboard API (not a target but supported for free).

Fallback contract: if `arboard::Clipboard::new()` returns an error (headless, no display, SSH),
the `ArboardClipboardAdapter::probe()` returns `ClipboardUnavailable`. The composition root sets
`AppConfig.clipboard_available = false`. In detail mode, pressing `c` shows the URL as
highlighted plain text with "Copy not available — select text manually." No crash, no panic.

`arboard` is added to `Cargo.toml` in slice-05. It is not imported in slice-01 through slice-04.

## Alternatives Considered

### Alternative 1: `clipboard` crate (rejected)

The `clipboard` crate (crates.io: `clipboard`) provides a similar API.

**Why rejected**:
- Last meaningful maintenance activity is several years old; the crate is in maintenance mode.
- Does not support Wayland natively — only X11 on Linux.
- `arboard` is the actively-maintained successor recommended by the Rust TUI community
  (ratatui ecosystem discussions, 2024–2025).
- `arboard` has more active contributors and a more recent release history.

### Alternative 2: Shell command fallback (`pbcopy` / `xclip` / `xsel`) (rejected)

Detect the OS and invoke `pbcopy` (macOS), `xclip`, or `xsel` (Linux) as subprocesses via
`std::process::Command`.

**Why rejected**:
- Requires subprocess execution, which introduces a new class of probe concerns (binary not in
  PATH, subprocess fails silently, output encoding issues).
- `arboard` already handles the platform differences natively — adding a subprocess fallback
  duplicates the platform-detection logic.
- `xclip` and `xsel` are not guaranteed to be installed; `arboard` bundles its own bindings.
- Shell-command clipboard is unreliable in headless environments by the same mechanism as
  library clipboard (no display server) — so the fallback provides no additional coverage.

### Alternative 3: No clipboard — URL display only (rejected)

Display the URL in the detail panel with no copy mechanism. User selects text manually.

**Why rejected**:
- US-08 is a "Should Have" story with a specific outcome KPI (< 5 seconds from detail view to
  URL in clipboard; zero mouse clicks). This alternative abandons the primary value of US-08.
- The fallback behaviour for SSH environments is already "display URL as text" — so the no-copy
  option is the fallback, not the primary. The primary must use arboard.

---

## Consequences

### Positive

- macOS clipboard copy works natively with NSPasteboard — the correct API for franci's primary
  environment.
- Linux (X11 and Wayland) supported without additional tooling.
- Graceful degradation on SSH/headless is built into the probe contract; no runtime surprises.
- `arboard` is MIT/Apache-2.0 licensed — compatible with the project's OSS-first policy.

### Negative (accepted trade-offs)

- `arboard` adds a native dependency on macOS (Objective-C bindings via `cc`). This increases
  compile time marginally. Build targets already include `aarch64-apple-darwin` — no new target
  risk.
- On Linux headless CI (GitHub Actions `ubuntu-latest`), the probe will return
  `ClipboardUnavailable`. Clipboard integration tests must be skipped or mocked in CI.
  Gold probe tests for clipboard must use `#[cfg(not(ci))]` or an environment flag.
- If `arboard` has a breaking API change, `ArboardClipboardAdapter` is the only affected file.
  The `ClipboardPort` trait insulates the rest of the codebase.

### Open Question

SSH clipboard forwarding detection: `arboard` will fail on SSH sessions without forwarding.
The probe catches this. However, if franci uses SSH with `-Y` (X11 forwarding), arboard may
succeed on an X11 forwarder that silently no-ops writes. A future spike (noted in
`feature-delta.md` open questions) should test arboard on SSH -Y and verify round-trip copy.
This does not block slice-05 on macOS.
