# ADR-002: Async Loading Strategy

**Status**: Accepted
**Date**: 2026-05-18
**Decider**: Morgan (Solution Architect) — approved by franci

---

## Context

rusty-commit-lister must scan the Obsidian vault at startup, parse daily note files, and render
the results in the TUI. The startup performance requirement is:

- First visible output (loading indicator): < 100ms
- Full table render: < 2 seconds for `scan_days_back` ≤ 30

`tokio` is already present in `Cargo.toml` from the bootstrap scaffold. The team (sole developer)
must decide how to structure the load path before writing slice-01 code.

Concrete data for the decision:
- `scan_days_back = 7` (default): 7 daily note files
- `scan_days_back = 30` (extended): up to 30 daily note files
- Each note file: ~2–5 KB, Markdown table parsing
- Estimated parse time per file: < 1ms (simple string scan, no regex)
- Total estimated scan time for 30 files: < 30ms on any modern machine
- Vault path involves Unicode emoji segment (spike test required in slice-01)

The design must also integrate with the Elm/MVU pattern (ADR-001): the scan result is fed to
`update()` as a `LoadComplete(Vec<CommitRecord>)` event, not set by direct mutation.

---

## Decision

**Synchronous blocking load for slice-01 and slice-02.**

The composition root (`main.rs`) calls `VaultScanPort::scan()` synchronously on the main thread
before starting the TUI event loop. The result is wrapped in `LoadComplete` or `LoadFailed` and
fed as the initial event to `update()` to seed the starting model.

The loading indicator is displayed by rendering the initial model with `AppMode::Loading` state,
which the TUI event loop shows on first `view()` call before the scan completes.

Wait — the sync call blocks before the TUI starts, so the spinner cannot render during the scan.
Correct sequencing for slice-01/02:

1. Composition root calls `scan()` (sync, < 30ms expected)
2. Scan returns `Vec<CommitRecord>` or error
3. Initial model is constructed with results
4. TUI event loop starts and renders the table immediately

The "loading indicator within 100ms" requirement is satisfied because the scan itself completes
in < 30ms. The spinner is shown in the initial render as a transitional state before the first
crossterm event tick — or elided entirely since the scan is fast enough to be invisible.

If observed startup latency exceeds 100ms on franci's real vault, this ADR is superseded by
ADR-002a which introduces async loading via channel.

---

## Options Evaluated

### Option A: Synchronous blocking load (chosen)

Load completes before TUI event loop starts. Simple. No channel, no task lifecycle.

**Trade-offs**:
- Pro: simplest implementation; no async complexity; model is fully populated on first render.
- Pro: zero risk of half-populated TUI state during load.
- Con: if scan takes > 100ms (unexpected), terminal appears frozen before TUI starts.
- Con: no progressive rendering (all-or-nothing).

**Risk mitigation**: spike test in slice-01 measures real vault scan time. If > 50ms, escalate
to Option B immediately.

### Option B: Async load with channel feeding LoadComplete event (deferred)

`tokio::task::spawn_blocking` runs the scan in a background thread. An `mpsc::channel` sends
`AppEvent::LoadComplete` or `AppEvent::LoadFailed` to the event loop. The event loop renders a
spinner until the event arrives.

**Trade-offs**:
- Pro: TUI renders spinner immediately; terminal is never frozen.
- Pro: supports progressive rendering if scan is split per-file.
- Con: adds `tokio::sync::mpsc` wiring, spawn_blocking task lifecycle, and channel poll in the
  event loop tick. Meaningful complexity increase for a 30ms operation.
- Con: initial model must handle `Loading` state explicitly in all view branches.

**Decision**: deferred to ADR-002a. Activate if Option A spike shows > 100ms latency.

### Option C: Stream results one-by-one to TUI (rejected)

Each parsed note file sends individual `NoteLoaded(Vec<CommitRecord>)` events as they complete.
The TUI renders partial results while scanning continues.

**Why rejected**:
- Significant complexity: requires streaming channel, partial state in AppModel, and a merge/sort
  step at the end (or per-event sort which is O(n log n) per file).
- The user gain is negligible: the scan window is small enough that even streaming would complete
  in < 100ms, indistinguishable from Option A or B.
- YAGNI until scan window exceeds 200+ notes.

---

## Consequences

### Positive

- Slice-01 implementation is simple: no async wiring, no channel, no task management.
- Domain model (`AppModel`) can start without a `Loading` variant in slice-01 (added later if
  Option B is needed).
- tokio runtime is available when needed for slice-05+ without any architectural change.

### Negative (accepted trade-offs)

- If the Unicode emoji vault path causes unexpected OsString overhead on macOS (known risk from
  DISCUSS wave), scan time could exceed expectations. Spike test in slice-01 is mandatory.
- "Loading indicator within 100ms" requirement is satisfied by the scan being fast, not by
  async rendering. If this assumption breaks, the design must change.

### Revision trigger

If the slice-01 spike test records vault scan > 100ms on franci's real machine
(`~/Documents/Wiki/📅 Diaries/0. Journal` with scan_days_back=7), create ADR-002a:
async load via `spawn_blocking` + `mpsc::channel`. The domain model and event enum do not
change — only the composition root wiring and the `Loading` AppMode variant are added.
