# ADR-003: Development Paradigm — Functional Programming vs OOP for Rust

**Status**: Accepted
**Date**: 2026-05-18
**Decider**: Morgan (Solution Architect) — approved by franci

---

## Context

Rust is a multi-paradigm language. It supports:
- Object-oriented patterns: structs with `impl` blocks, trait objects, inheritance via trait composition, mutable self-methods, interior mutability (`RefCell`, `Mutex`).
- Functional patterns: pure functions, immutable data by default, algebraic data types (enums), iterator combinators, closures, pattern matching, type-driven design.

The development team is a sole developer (franci). There is no team paradigm convention to
align with. The paradigm choice affects:
- How the state machine is structured (ADR-001 already decided Elm/MVU — this ADR formalises the
  broader paradigm commitment that ADR-001 implies).
- How the parser is written (`parse_note` as pure function vs method on a struct with mutable state).
- How adapters are designed (stateless structs with function methods vs mutable objects).
- What future contributors (or AI crafters) are directed to use.
- Whether tests require setup/teardown or are pure function calls.

The Elm/MVU decision (ADR-001) already commits the domain to pure functions. This ADR extends
that commitment to the whole codebase and documents it explicitly for the CLAUDE.md handoff to
`/nw-develop`.

---

## Decision

**Functional Programming (FP) paradigm throughout the codebase.**

Concrete commitments:
1. **Pure functions first**: prefer `fn foo(input: X) -> Y` over `fn foo(&mut self, ...)`.
2. **Immutable data by default**: `let` not `let mut`; mutability added only where genuinely
   required (e.g., `Frame` during render, `Vec` during construction).
3. **Types-first design**: define data types (`CommitRecord`, `AppModel`, `AppEvent`) before
   writing functions. Let the type signatures drive the implementation shape.
4. **Algebraic data types for state**: `AppMode` as enum, not as a class hierarchy or flag fields.
5. **Iterator combinators over explicit loops**: `.filter()`, `.map()`, `.collect()` over `for`
   loops with mutation where semantically equivalent.
6. **No interior mutability in domain**: `RefCell`, `Mutex`, `Cell` are forbidden in
   `src/domain/`. Adapters may use them internally if required by a library (e.g., arboard).
7. **Composition over inheritance**: no trait object hierarchies for domain logic; prefer concrete
   types with port traits at boundaries.

CLAUDE.md at the project root will direct `/nw-develop` and future AI crafters to use
`@nw-functional-software-crafter`.

## Alternatives Considered

### Alternative 1: OOP / Mutable App Struct (rejected)

All state lives in a single `App` struct with mutable fields. Methods on `App` handle events by
mutating state in place. Structs for parser (`MarkdownParser`), config loader (`ConfigLoader`),
etc., with `new()` constructors and method chains.

**Why rejected**:
- The mutable App struct pattern was already evaluated and rejected in ADR-001 for the TUI state
  machine. Extending OOP to the rest of the codebase would contradict ADR-001's rationale.
- Parser as a struct with mutable state adds no value: `parse_note()` has no persistent state
  between calls. A struct constructor would be boilerplate for no benefit.
- OOP tests require construction (`App::new()`) with all collaborator fields set. FP tests are
  pure function calls. The sole-developer context makes test simplicity a high-value property.
- Rust's ownership model actively discourages shared mutable state — FP aligns with Rust's
  defaults rather than fighting them.

### Alternative 2: Mixed Paradigm (rejected)

FP for domain and parser; OOP with mutable structs for adapters and TUI.

**Why rejected**:
- The boundary between "domain" and "adapter" is already enforced by port traits. Within
  adapters, FP methods are just as natural (an adapter `write(text: &str) -> Result<()>` is a
  pure-ish method regardless of paradigm label).
- A mixed-paradigm project requires the developer to context-switch between styles within the
  same session. For a sole developer building a personal tool, a single consistent style reduces
  cognitive load.
- The distinction is not meaningful for this codebase size. FP everywhere is simpler than "FP
  here, OOP there" rules.

---

## Consequences

### Positive

- Tests for domain and parser are trivial pure-function calls. No test setup, no mocks required
  for the core logic.
- State bugs are structurally constrained: `update()` returning a new model means "partially
  updated model" is not a valid state.
- Code is more readable for a sole developer who reasons in terms of data transformation
  (what goes in, what comes out) rather than object lifecycle.
- `@nw-functional-software-crafter` directive in CLAUDE.md ensures future AI-assisted
  development sessions apply the same paradigm without re-negotiating.

### Negative (accepted trade-offs)

- Some Rust ecosystem libraries expect mutable self receivers (e.g., arboard, crossterm raw mode
  setup). These are contained in adapters; the domain is not affected.
- Iterator combinators can be less readable than explicit loops for developers unfamiliar with
  Rust idioms. Risk is low: sole developer chose this paradigm and is comfortable with it.
- `AppModel` derives `Clone` for ownership threading through `update()`. This is a minor
  performance cost for a data structure of this size (< 10KB).

### Handoff

`CLAUDE.md` at the project root documents this decision and directs implementation to
`@nw-functional-software-crafter`. This ADR is the normative source of truth; CLAUDE.md
is the operational directive.
