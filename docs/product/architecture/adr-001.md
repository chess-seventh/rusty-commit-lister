# ADR-001: TUI Architecture — Elm/MVU vs Mutable App Struct

**Status**: Accepted
**Date**: 2026-05-18
**Decider**: Morgan (Solution Architect) — approved by franci

---

## Context

rusty-commit-lister requires a ratatui TUI with four distinct interaction modes:

- **Browse**: navigable commit table (default mode)
- **Search**: inline search bar active; keystrokes go to query string
- **Detail**: commit detail overlay panel open
- **RepoPicker**: repo filter overlay open

These four modes compose. Search and RepoPicker can be active simultaneously (AND-filter logic).
The Esc key has different semantics depending on active mode (close panel, clear search, or exit).

The TUI state includes:
- Loaded commit records (`Vec<CommitRecord>`)
- Current row cursor index
- Active search query string
- Active repo filter
- Detail panel visibility
- Clipboard operation result (confirmation or error)
- Loading state (before first scan completes)

The ratatui ecosystem supports two broad approaches to managing this state.

The design must also satisfy:
- Testability requirement: state transitions must be unit-testable without a terminal.
- Sole developer: no team coordination overhead.
- 6-slice roadmap: state machine will grow across slices (Browse in slice-01, Search in slice-03, Detail in slice-04, RepoPicker in slice-06).

---

## Decision

**Elm/MVU architecture**:

```
update(model: AppModel, event: AppEvent) -> AppModel
view(model: &AppModel, frame: &mut Frame)
```

`update()` is a **pure function**. It takes ownership of the current model and an event, and
returns a new model. It has no side effects. All four modes are variants of a single `AppMode`
enum. State transitions are pattern-matched in a single `update()` function.

`view()` is a **pure render function**. It reads `&AppModel` and writes to `&mut Frame`. No
state mutation, no side effects.

Effects (vault scan, clipboard write) are pushed to the event loop shell layer. They produce
`AppEvent` values (`LoadComplete`, `ClipboardResult`) that re-enter `update()`. The domain
never calls I/O directly.

---

## Alternatives Considered

### Alternative 1: Mutable App Struct (rejected)

A single `App` struct holds all state as mutable fields. The event loop calls `app.handle_event(event)` which mutates fields in place. This is the most common pattern in ratatui tutorial code.

**Why rejected**:
- Cross-mode mutation is harder to trace: `handle_event()` becomes a large match that mutates
  different fields depending on mode, with implicit coupling between branches.
- Testing requires constructing the full `App` struct with all fields set to known values, then
  calling `handle_event()` and asserting field values — setup-heavy and brittle.
- Adding a new mode (e.g., RepoPicker in slice-06) requires auditing all existing branches for
  mode-leakage bugs (e.g., j/k applying in RepoPicker when it should not).
- Ownership semantics work against mutation: ratatui requires `&mut Frame` during `view()`, which
  conflicts with holding mutable references to state in the same scope.

### Alternative 2: Actor/Channel-Based State (rejected)

Each mode runs as a separate Tokio task communicating via channels.

**Why rejected**:
- Introduces async complexity (tokio, channels, task lifecycle) for a problem that is solved
  purely by a function. The concurrency model is not needed for the state machine itself.
- Debugging state bugs requires tracing channel messages across task boundaries.
- Single-developer tool with a simple state machine does not warrant actor-model overhead.
- YAGNI: the async upgrade path (ADR-002) covers the only genuine async requirement (vault scan).

---

## Consequences

### Positive

- State corruption bugs are structurally impossible: `update()` produces a new model; it never
  partially mutates an existing one.
- Every state transition is a pure function call — trivially unit-testable with no I/O, no setup,
  no mocks. Test = `assert_eq!(update(input_model, event), expected_model)`.
- Four-mode enum maps cleanly to nested `match (model.mode, event)` arms. Adding RepoPicker
  in slice-06 adds new arms without touching existing Browse/Search/Detail arms.
- `view()` pure render means the visual output is a deterministic function of the model — no
  display bugs from shared display state.
- The Elm/MVU pattern is well-documented and widely understood; a future contributor can read
  the domain in isolation without understanding ratatui internals.

### Negative (accepted trade-offs)

- Model ownership must be passed through `update()` on every event, even for events that change
  only one field. In Rust, this requires either `Clone` on `AppModel` or careful ownership
  threading. Decision: `AppModel` derives `Clone`; the event loop does `let model = update(model, event)`.
  The clone cost is negligible for this data size (< 10KB of state including commit records).
- Async effects (clipboard, future async scan) feed events via channel to the event loop rather
  than being called inline. The event loop must poll the channel on each tick. This is a minor
  structural constraint with negligible runtime cost.
- Rust's ownership model means `AppModel` fields containing `Vec<CommitRecord>` are cloned on
  every update. Mitigation: `Arc<Vec<CommitRecord>>` for the commit list if profiling shows
  clone cost above 1ms. This is deferred to slice-02 only if needed.
